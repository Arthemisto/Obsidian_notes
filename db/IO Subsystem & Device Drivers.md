## 🔌 [[IO Subsystem & Device Drivers]]

> **Зачем это тут:** после **memory + storage hierarchy** процессам нужно читать/писать данные и общаться с устройствами. ОС скрывает “зоопарк железа” за единым интерфейсом: **I/O subsystem + device drivers**.

---
## 🎛️ Роль I/O Subsystem — “прятать железо” (hide hardware peculiarities)
Одна из целей ОС (**OS**) — скрывать особенности аппаратных устройств (**hardware devices**) от пользователя (**user**) и приложений, чтобы доступ к диску/сети/клавиатуре выглядел единообразно.

I/O subsystem отвечает за:
- **унификацию доступа** к устройствам (общий подход вместо “каждое устройство по-своему”)
- **производительность и стабильность** работы ввода-вывода
- связку: **processes ↔ drivers ↔ devices**

---
## 🧩 Buffering / Caching / Spooling — базовые техники ускорения
I/O subsystem управляет “памятью ввода-вывода” (**memory management of I/O**) через:

- **buffering** — временное хранение данных при передаче (сглаживает разницу скоростей CPU ↔ device)
- **caching** — хранение части данных в более быстром хранилище для ускорения повторного доступа
- **spooling** — постановка/очередь на устройство, чтобы перекрывать вывод одной задачи вводом/работой других (классика: печать)

> Мостик: эти техники нужны, потому что устройства работают медленно и неравномерно по сравнению с CPU/ RAM.

---
## 🧩 Device-driver interface + drivers — “переводчик” между ОС и устройством
- **device-driver interface** — общий интерфейс, через который ОС обращается к классам устройств.
- **device drivers** — конкретная реализация под железо (**specific hardware devices**): переводит общие команды ОС в операции контроллера устройства.

Зачем драйверы:
- изоляция особенностей конкретного устройства
- возможность менять устройство без переписывания приложений (в идеале)

---
## 📥📤 Handle Input/Output Operations / Операции ввода-вывода
ОС обрабатывает **input/output operations**, включая взаимодействие с:
- **storage devices** (диски/SSD)
- **network interfaces** (сеть)
- другой периферией (**peripherals**)

> Мостик: как только есть процессы + файлы + устройства, возникает вопрос “кто что может делать” → нужен контроль доступа.

---
## 🧭 Summary / Сводка
- I/O subsystem скрывает особенности устройств и делает доступ единообразным.
- Производительность I/O держится на **buffering / caching / spooling**.
- Драйверы = связка **OS ↔ hardware** через **device-driver interface**.
